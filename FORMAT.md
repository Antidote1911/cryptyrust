# Cryptyrust encrypted file format (v1.1.0)

> **Compatibility:** v1.1.0 files are not compatible with v1.0.0. The full header is now used as Additional Authenticated Data (AAD) — see below.

## Binary format (`.crypty`)

The file starts with the magic number `43 52 59 50`, which lets Cryptyrust detect its own files and automatically switch to decryption mode.

### Header layout (64 bytes total)

| Field | Size | Values |
|---|---|---|
| Magic number | 4 bytes | `43 52 59 50` |
| Header version | 2 bytes | `DE 01` → Version 1 |
| Algorithm | 2 bytes | `0E 01` XChaCha20Poly1305 / `0E 02` AES-256-GCM / `0E 03` AES-256-GCM-SIV |
| Argon2 strength | 2 bytes | `BE 01` Interactive / `BE 02` Moderate / `BE 03` Sensitive |
| Salt | 16 bytes | Random |
| Reserved | 16 bytes | `00 00 … 00` (future use) |
| Nonce | 8 or 20 bytes | Random (8 for AES, 20 for XChaCha20Poly1305) |
| Padding | 14 or 2 bytes | `00 00 … 00` (fills header to exactly 64 bytes) |

Nonces are 4 bytes shorter than the standard algorithm nonce size because STREAM mode (EncryptorLE31 / DecryptorLE31) reserves the last 4 bytes for a little-endian counter incremented after each chunk.

| Algorithm | Standard nonce | Stored nonce | Header padding |
|---|---|---|---|
| AES-256-GCM | 12 bytes | 8 bytes | 14 bytes |
| AES-256-GCM-SIV | 12 bytes | 8 bytes | 14 bytes |
| XChaCha20Poly1305 | 24 bytes | 20 bytes | 2 bytes |

### Encrypted data

After the 64-byte header, the file contains a sequence of encrypted chunks:

```
[ BUFFER_SIZE bytes of plaintext ] → [ BUFFER_SIZE + 16 bytes ciphertext + auth tag ]
```

`BUFFER_SIZE` = 10 × 1024 bytes (10 KB).

### Additional Authenticated Data (AAD)

The entire 64-byte header is passed as AAD to the AEAD stream cipher. Any modification to the header after encryption causes decryption to fail with an authentication error.

## PEM format (`.crypty.pem`) — CLI and GUI

The GUI can optionally output the same binary structure encoded as a PEM-style text file. The binary content (header + encrypted chunks) is base64-encoded in 64-character lines:

```
# Generated by Cryptyrust v1.1.0
-----BEGIN CRYPTYRUST ENCRYPTED DATA-----
Q1JZUН4BDgK+AxdbAA...
...
-----END CRYPTYRUST ENCRYPTED DATA-----
```

- Encoding: standard Base64 (RFC 4648), 48 bytes of binary per line → 64 base64 characters
- Detection: Cryptyrust recognises PEM files by scanning the first few lines for the `-----BEGIN CRYPTYRUST ENCRYPTED DATA-----` marker — detection is content-based, not extension-based
- CLI: use `--pem` to encrypt to PEM format; decryption is always auto-detected
- The internal binary structure (header + AAD + chunks) is identical to the `.crypty` binary format
