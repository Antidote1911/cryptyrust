Arsenic V1

CRYPTYRUST FILE FORMAT SPECIFICATION  ·  MAGIC 41 52 53 4E  ·  VERSION 00 01  ·  EXTENSION .arsn

1. Overview & Design Principles

An Arsenic V1 file consists of a variable-length header followed by a sequence of independently authenticated payload blocks. Every structural decision below is justified by what it provides.

  • DEK separation — the Data Encryption Key is 32 random bytes wrapped under the password-derived KEK. Changing a password never re-encrypts the payload.
  • LUKS-style keyslot — the 48-byte WrappedDEK is the only section modified on rekey. Metadata is bound to the DEK, not to the password.
  • Tiny Argon2id pre-auth — a cheap KDF pass (~2 ms) produces a pre-authentication key for the header MAC. Wrong passwords are rejected quickly without paying the full KDF cost, and without exposing a fast offline oracle.
  • Full parallelism — per-block key and nonce derivation are independent, so encryption and decryption scale linearly with CPU core count.
  • Merkle v1 integrity — domain-separated BLAKE3 tree over all encrypted blocks; no plaintext is written until the whole file passes verification.

3. Header Layout — Precise Byte Table

The header has a variable total size stored at offset 0x0A as a little-endian u16. The minimum is 232 bytes (mandatory TLV only, no optional fields).

3.2 Header MAC (bytes 0x04C–0x06B, 32 bytes)

OffsetSizeFieldDerivation
0x04C32HeaderMACHMAC-SHA256(PreKey, header[0x000..0x04C])

PUB_HEADER_LEN = 108 bytes (public section + MAC).

3.3 WrappedDEK — keyslot (bytes 0x06C–0x097, 48 bytes, fixed)

WrappedDEK = AEAD_hdr_cipher(
    key   = KEK,                    ← 32-byte key, Argon2id-derived
    nonce = derived from kek_nonce, ← see §7 for per-cipher expansion
    aad   = empty,
    msg   = DEK                     ← 32 random bytes, generated at encryption time
)
→ [32 bytes ciphertext][16 bytes AEAD tag] = 48 bytes total
Why a dedicated keyslot? The keyslot model (LUKS / BitLocker / VeraCrypt) decouples the password from the payload. On a password change, only the 48-byte keyslot is re-encrypted with the new KEK. The DEK — and therefore the entire payload and all metadata — is unchanged.

3.4 ProtectedMetadata (bytes 0x098–…, variable)

MetaKey   = BLAKE3_derive_key("Arsenic V1 Metadata Key", DEK)       → 32 bytes
MetaNonce = BLAKE3_derive_key("Arsenic V1 Meta Nonce",   DEK)[0..12] → 12 bytes

ProtectedMetadata = AEAD_hdr_cipher(
    key   = MetaKey,
    nonce = MetaNonce,
    aad   = empty,
    msg   = TLV_bytes
)
→ [len(TLV_bytes) ciphertext][16 bytes AEAD tag]
Why MetaKey = f(DEK)? MetaKey is derived from the DEK, not from the password. Since the DEK never changes (even across rekeys), the encrypted ProtectedMetadata bytes are stable. Rekey copies them byte-for-byte without any decryption or re-encryption.

3.5 TLV encoding in ProtectedMetadata

Each TLV record: [tag: 1 byte][length: 1 byte][value: length bytes]. Max value size = 255 bytes. First occurrence wins on duplicates. Unknown tags are silently skipped (forward compatibility).

4.2 Per-block key and nonce derivation

BlockKey_N   = BLAKE3_keyed_hash(
    key  = DEK,
    data = u64_LE(N)
) → 32 bytes

BlockNonce_N = BLAKE3_derive_key(
    context = "Arsenic V1 Block Nonce",
    data    = file_base_nonce ‖ u64_LE(N)   (24 + 8 = 32 bytes)
)[0..24] → 24 bytes
           first 12 bytes used by AES-256-GCM-SIV  (96-bit nonce)
           first 15 bytes used by Deoxys-II-256     (120-bit nonce)
           all 24 bytes used by XChaCha20-Poly1305  (192-bit nonce)
Each block's key and nonce depend only on the DEK, block index, and file_base_nonce. They are mutually independent, enabling full Rayon parallelism over all blocks during both encryption and decryption.

4.3 Block encryption — uncompressed (compression_id = 0x00)

EncBlock_N = PayloadCipher(
    key   = BlockKey_N,
    nonce = BlockNonce_N[0..12]   (AES-256-GCM-SIV)
          | BlockNonce_N[0..15]   (Deoxys-II-256)
          | BlockNonce_N[0..24]   (XChaCha20-Poly1305),
    aad   = u64_LE(N),            ← block index binds position
    msg   = plaintext_block_N
) → [plaintext_len + 16] bytes

Block count: num_blocks = compressed_size ÷ block_size (ceiling division). The last block may be shorter than block_size.

4.4 Block encryption — zstd per-block (compression_id = 0x01)

CompressedBlock_N = zstd_compress(plaintext_block_N, level=3)

EncBlock_N = PayloadCipher(
    key   = BlockKey_N,
    nonce = BlockNonce_N[0..12 | 0..15 | 0..24],   ← depends on cipher (see §4.2)
    aad   = u64_LE(N),
    msg   = CompressedBlock_N
) → [compressed_len + 16] bytes

On-disk:  [enc_size: u32 LE][EncBlock_N: enc_size bytes]
          where enc_size = compressed_len + 16

Block count: num_blocks = original_size ÷ block_size (ceiling division, using the uncompressed size because original blocks are split before compression).

Why per-block? Whole-file compression buffers the entire file in RAM before splitting. Per-block compression keeps memory at O(block_size) ≤ 32 MiB, preserves Rayon parallelism (each block is an independent compress → encrypt unit), and the u32 size prefix makes the decoder position-independent after a sequential pre-scan.

5. Header Pre-Authentication

PreKey    = Argon2id(
    password = user passphrase (UTF-8),
    salt     = header[0x00C..0x01C],   ← 16 bytes
    t        = 1,
    m        = 8 192 KB,              ← 8 MiB
    p        = 1,
    taglen   = 32
) → 32 bytes

HeaderMAC = HMAC-SHA256(
    key  = PreKey,
    data = header[0x000..0x04C]        ← all 76 public bytes
) → 32 bytes
SchemeGPU H/s (RTX 4090)vs raw HMAC
Raw HMAC-SHA256(password, salt)~20 000 000 000oracle — Argon2id completely bypassed
Tiny Argon2id (m = 8 MB)~15 000×1 300 000 improvement
Full Interactive KEK (m = 256 MB)~100—
A raw HMAC over the password would turn the HeaderMAC into a fast offline oracle. Any attacker with the .arsn file could bypass Argon2id entirely and test billions of passwords per second. The tiny Argon2id pre-key eliminates this while keeping wrong-password rejection at ~2 ms.

7. Cipher Algorithms

IDAlgorithmNonce bitsTag bytesNotes
0x02Deoxys-II-25612016Tweakable-block-cipher AEAD (RustCrypto). Default header cipher.
0x03XChaCha20-Poly130519216192-bit nonce eliminates nonce-collision risk. Default payload cipher.
0x04AES-256-GCM-SIV9616Nonce-misuse resistant: safe if a nonce is accidentally reused.

The kek_nonce field is always 12 bytes. Each cipher derives its actual nonce from it via BLAKE3:

— AES-256-GCM-SIV (96-bit nonce) —
nonce = kek_nonce   ← used directly, no expansion needed

— Deoxys-II-256 (120-bit nonce) —
nonce15 = BLAKE3_derive_key(
    "Arsenic V1 KEK Nonce DeoxysII256",
    kek_nonce ‖ 0x00×20            ← zero-padded to 32 bytes
)[0..15]

— XChaCha20-Poly1305 (192-bit nonce) —
nonce24 = BLAKE3_derive_key(
    "Arsenic V1 KEK Nonce XChaCha20",
    kek_nonce ‖ 0x00×20            ← zero-padded to 32 bytes
)[0..24]
Performance note — header vs payload cipher: the header cipher encrypts only 32 bytes (the DEK in the WrappedDEK keyslot), regardless of file size. This operation takes nanoseconds and its choice has no measurable impact on throughput. The payload cipher processes the entire file in 4–32 MiB blocks and is the sole determinant of encryption speed. The built-in cipher benchmark therefore measures payload cipher throughput only and recommends the same cipher for both roles.

8. Merkle Tree v1 — BLAKE3 Domain-Separated

Leaf_N     = BLAKE3_derive_key("Arsenic V1 Merkle Leaf v1",  EncBlock_N)
Node(L, R) = BLAKE3_derive_key("Arsenic V1 Merkle Node v1",  L ‖ R)
             where L, R are 32-byte child hashes (big-endian order: left child first)

Empty file (0 blocks):  root = [0x00 × 32]
Single block:           root = Leaf_0
Multiple blocks:
  Pair adjacent leaves/nodes → Node(left, right)
  If count is odd, the last node is promoted unchanged (v1 promotion rule)
  Repeat until one node remains → Merkle root
Why domain separation? Without it, a 64-byte block B = left ‖ right satisfies BLAKE3(B) = Node(left, right) — a second-preimage attack. BLAKE3_derive_key puts leaves and nodes in disjoint output domains, making this impossible regardless of block content. The promotion of odd nodes is safe for the same reason: a promoted leaf hash cannot be confused with a node hash.

The Merkle root is stored at TLV tag 0x02 inside ProtectedMetadata. On decryption, all blocks are decrypted in parallel (Rayon), the root is recomputed from the resulting ciphertexts, and compared before any plaintext is written to disk.

9. Password Change (Rekey)

Rekey is a LUKS-style keyslot replacement. Only the 48-byte WrappedDEK changes:

  1. Read the full header (header_total_size bytes).
  2. Run tiny Argon2id → PreKey → verify HeaderMAC with old password.
  3. Derive old KEK with full Argon2id.
  4. Decrypt WrappedDEK with old KEK → recover the 32-byte DEK.
  5. Generate fresh new_salt (16 bytes) and new_kek_nonce (12 bytes).
  6. Derive new KEK = Argon2id(new_password, new_salt, same t/m/p).
  7. Encrypt DEK with new KEK → new_WrappedDEK (48 bytes).
  8. Assemble: updated public section + new HeaderMAC + new_WrappedDEK + original ProtectedMetadata bytes copied unchanged.
  9. Write the new header in-place (same header_total_size).

Unchanged after rekey: DEK · Merkle root · payload · metadata · cipher IDs · file_base_nonce · header_total_size.

ProtectedMetadata does not need re-encryption because MetaKey = BLAKE3_derive_key("Arsenic V1 Metadata Key", DEK). The DEK is identical after rekey, so MetaKey is identical, and the existing ciphertext bytes remain valid.

Crash safety

A backup of the current header (<file>.bak) is written and sync_all-flushed before the in-place write. On success the backup is deleted. If interrupted:

  • Magic intact → write completed; backup is stale.
  • Magic corrupted → write was interrupted; backup is used to restore, error returned.
