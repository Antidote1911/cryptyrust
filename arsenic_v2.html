Protocole Arsenic V2

Spécification Technique de Référence — Chiffrement de Fichiers à Haute Sécurité
Auteur : Fabrice Corraire <antidote1911@gmail.com>
Statut : Spécification Finale — format .arsn
Focus Opérationnel : Algorithmes sélectionnables indépendamment (enveloppe et payload), résistance DoS mémoire, intégrité Merkle, parallélisme Rayon.

1. Vue d'ensemble de l'Architecture

La version V2 du protocole Arsenic est conçue pour le chiffrement de données hautement confidentielles. Elle utilise un paradigme d'exécution en deux passes permettant de figer l'intégrité de l'intégralité du fichier via un arbre de Merkle BLAKE3 avant de sceller définitivement son en-tête.

Sélection Indépendante des Algorithmes

Une innovation clé de Arsenic V2 est la possibilité de choisir indépendamment deux algorithmes AEAD distincts :

  • Header Cipher (octet 0x07) — chiffre l'enveloppe de clé DEK dans l'en-tête.
  • Payload Cipher (octet 0x08) — chiffre chaque bloc de données du fichier.

Les deux IDs sont stockés en clair dans la section pré-MAC et sont couverts par le HeaderMAC. Ils ne peuvent pas être falsifiés sans invalider l'authentification.

Un fichier Arsenic V2 est structuré comme suit :

┌──────────────────────────────────────────────┐
│ En-tête — 256 octets │
│ ├─ Section publique pré-MAC (76 octets) │
│ ├─ HeaderMAC HMAC-SHA256 (32 octets) │
│ ├─ Enveloppe chiffrée (97 octets) │ ← header cipher (sélectionnable)
│ └─ Padding zéro (51 octets) │
├──────────────────────────────────────────────┤
│ Bloc chiffré 0 │ ← payload cipher (sélectionnable)
├──────────────────────────────────────────────┤
│ Bloc chiffré 1 │
├──────────────────────────────────────────────┤
│ … │
└──────────────────────────────────────────────┘

2. Structure de l'En-tête (256 octets)

L'en-tête est aligné sur 256 octets pour des performances d'E/S optimales. Le champ Header_Total_Size permet une évolution future sans rupture de format.

2.1 Section Publique Pré-MAC (octets 0x00–0x4B, 76 octets)

Offsets Taille Champ État Description
0x00–0x03 4 octets Magic Number En Clair Signature immuable : 0x41 0x52 0x53 0x4E ("ARSN").
0x04–0x05 2 octets Version Info En Clair 0x00 0x02 — Arsenic V2.
0x06 1 octet KDF_ID En Clair Identifiant KDF : 0x01 = Argon2id.
0x07 1 octet Header_Cipher_ID En Clair Algorithme de l'enveloppe de clé — sélectionnable :
0x02 = Serpent-256-GCM (défaut)
0x03 = XChaCha20-Poly1305
0x04 = AES-256-GCM-SIV
Couvert par le HeaderMAC.
0x08 1 octet Payload_Cipher_ID En Clair Algorithme des blocs de données — sélectionnable :
0x02 = Serpent-256-GCM
0x03 = XChaCha20-Poly1305 (défaut)
0x04 = AES-256-GCM-SIV
Couvert par le HeaderMAC.
0x09 1 octet Compression_ID En Clair Algorithme de compression : 0x00 = Aucun.
0x0A–0x0B 2 octets Header_Total_Size En Clair u16 LE — taille totale de l'en-tête. Défaut : 0x0100 = 256 octets. Permet une évolution future sans rupture.
0x0C–0x1B 16 octets Argon2id Salt En Clair Sel cryptographique aléatoire (OsRng). Base de la PreKey.
0x1C–0x1F 4 octets t_cost En Clair Itérations Argon2id (u32 LE). Couvert par le HeaderMAC — ne peut pas être dégradé silencieusement.
0x20–0x23 4 octets m_cost En Clair Mémoire Argon2id en Ko (u32 LE). Validé après vérification du HeaderMAC pour éviter le DoS.
0x24–0x27 4 octets p_cost En Clair Parallélisme Argon2id (u32 LE).
0x28–0x3F 24 octets File Base Nonce En Clair Nonce maître aléatoire. Sert de base au déterminisme des nonces de blocs via BLAKE3.
0x40–0x4B 12 octets KEK Nonce En Clair Nonce d'initialisation pour le chiffrement de l'enveloppe (12 octets stockés). Pour XChaCha20-Poly1305, étendu à 24 octets via BLAKE3_derive_key — voir Section 3.2.
0x4C–0x6B 32 octets Header MAC En Clair VÉRIFIÉ EN PREMIER. HMAC-SHA256 sur les octets 0x00–0x4B via la PreKey. Neutralise toute tentative de DoS par m_cost falsifié.

2.2 Enveloppe Chiffrée (octets 0x6C–0xC2, 97 octets)

L'enveloppe est chiffrée par le header cipher identifié en octet 0x07, en utilisant la KEK dérivée d'Argon2id. Tous les algorithmes supportés produisent un tag AEAD de 16 octets, donc l'enveloppe chiffrée est toujours exactement 97 octets (81 octets de plaintext + 16 octets de tag).

Offset (dans le plaintext) Taille Champ
0 32 octets DEK — Data Encryption Key, aléatoire, unique par fichier.
32 32 octets Merkle Root — racine BLAKE3 de l'arbre d'intégrité sur les blocs chiffrés.
64 8 octets Original Size — taille originale du fichier (u64 LE).
72 8 octets Compressed Size — taille du payload compressé (u64 LE).
80 1 octet Block Size ID — 0x01 = blocs 4 Mio (fichiers < 4 Gio) · 0x02 = blocs 32 Mio (fichiers ≥ 4 Gio).

2.3 Padding (octets 0xC3–0xFF, 51 octets)

Octets zéro, réservés pour les évolutions futures.

3. Algorithmes Supportés et Gestion des Nonces

3.1 Tableau des Algorithmes

ID Algorithme Nonce Notes
0x02 Serpent-256-GCM 96 bits (12 oct.) Serpent-256 en mode GCM NIST ; implémentation GHASH manuelle. Algorithme d'enveloppe par défaut.
0x03 XChaCha20-Poly1305 192 bits (24 oct.) Nonce étendu à 192 bits — risque de collision de nonce quasi nul à grande échelle. Algorithme de payload par défaut.
0x04 AES-256-GCM-SIV 96 bits (12 oct.) Variante GCM résistante à la réutilisation de nonce. Sûr même en cas de nonce accidentellement répété.

Les trois algorithmes produisent un tag d'authentification de 16 octets. La taille du ciphertext est toujours plaintext_len + 16, quel que soit l'algorithme.

3.2 Gestion du Nonce KEK pour l'Enveloppe

Le champ KEK_nonce est toujours stocké sur 12 octets en en-tête. Pour Serpent-256-GCM et AES-256-GCM-SIV (nonces 96 bits), ce champ est utilisé directement. Pour XChaCha20-Poly1305 qui nécessite 192 bits, le nonce est étendu déterministiquement par BLAKE3 :

// KEK nonce stocké = 12 octets
// Pour Serpent-256-GCM et AES-256-GCM-SIV : utilisation directe des 12 octets

// Pour XChaCha20-Poly1305 en enveloppe :
input_32 = KEK_nonce[0..12] ‖ 0x00×20    // zéro-pad à 32 octets
nonce_24 = BLAKE3_derive_key("Arsenic V2 KEK Nonce XChaCha20", input_32)[0..24]

3.3 Dérivation des Clés et Nonces de Blocs (BLAKE3)

Chaque bloc dispose d'une clé et d'un nonce dérivés déterministiquement depuis la DEK et l'index du bloc, permettant le parallélisme total sans synchronisation :

// Clé de bloc — BLAKE3 keyed hash
BlockKey_N = BLAKE3_keyed_hash(key = DEK, data = u64_LE(N))

// Nonce de bloc — BLAKE3 derive_key (toujours 24 octets dérivés)
BlockNonce_N = BLAKE3_derive_key("Arsenic V2 Block Nonce",
                      File_Base_Nonce ‖ u64_LE(N))[0..24]

// Pour les ciphers à nonce 96 bits (Serpent-GCM, AES-GCM-SIV) :
nonce_utilisé = BlockNonce_N[0..12]

// Pour XChaCha20-Poly1305 : BlockNonce_N[0..24] utilisé en entier

4. Pipeline Cryptographique

4.1 Pré-authentification et Immunité DoS

Avant d'allouer la moindre structure mémoire pour Argon2id, une validation bon marché rejette immédiatement toute en-tête falsifiée :

// 1. Dérivation de la PreKey (coût : un seul appel HMAC)
PreKey = HMAC-SHA256(Key = Password, Data = Header.Salt)

// 2. Vérification en temps constant du MAC
Computed_MAC = HMAC-SHA256(Key = PreKey, Data = Header[0x00..0x4B])
if ConstantTimeCompare(Computed_MAC, Header.Header_MAC) != Success {
    Abort("Authentification rejetée — mauvais mot de passe ou en-tête corrompu");
}

// 3. Validation des limites de sécurité (après vérification MAC)
if Header.m_cost > MAX_ARGON2_RAM_KB {
    Abort("Paramètres hors limites de sécurité");
}

4.2 Dérivation de la KEK via Argon2id

KEK = Argon2id(
    password = mot de passe UTF-8,
    salt = Header[0x0C..0x1C]  // 16 octets aléatoires,
    t = t_cost,
    m = m_cost (Ko),
    p = p_cost,
    taglen = 32
)
Préréglage t m (Ko) p Mémoire Durée typique
Interactive (défaut) 4 262 144 4 256 Mio ~1–3 s
Sensitive 12 1 048 576 4 1 Gio ~10–30 s

4.3 Passe 1 : Fragmentation et Chiffrement Parallèle des Blocs

  1. Segmentation adaptative :
    • Fichiers < 4 Gio : blocs de 4 Mio (Block ID 0x01)
    • Fichiers ≥ 4 Gio : blocs de 32 Mio (Block ID 0x02)
  2. Chiffrement indépendant par bloc (parallèle via Rayon) : Chaque thread dérive sa BlockKey_N et son BlockNonce_N depuis la DEK, puis chiffre avec le payload cipher sélectionné. L'index du bloc est injecté dans les données authentifiées supplémentaires :
    EncBlock_N = PayloadCipher(key=BlockKey_N, nonce=BlockNonce_N, aad=u64_LE(N), msg=plaintext_N)
  3. Calcul des feuilles Merkle : L'empreinte de chaque bloc chiffré est calculée et conservée en RAM :
    Leaf_N = BLAKE3(EncBlock_N)

4.4 Passe 2 : Clôture de l'Arbre de Merkle et Scellement de l'En-tête

  1. Calcul de la racine Merkle : Les feuilles sont combinées récursivement par paires — les nœuds impairs sont promus sans hachage — produisant la Merkle Root de 32 octets. Elle scelle mathématiquement l'ordre, l'intégrité et le nombre exact de blocs.
  2. Construction de l'enveloppe : DEK, Merkle Root, tailles et Block Size ID sont sérialisés (81 octets).
  3. Scellement de l'enveloppe : La zone confidentielle est chiffrée avec le header cipher sélectionné via la KEK issue d'Argon2id.
  4. Calcul du HeaderMAC et écriture de l'en-tête complet (256 octets) en début de fichier.

La DEK est générée aléatoirement à chaque chiffrement. Un changement de mot de passe (rekey) re-chiffre uniquement l'enveloppe sous une nouvelle KEK — le payload n'est jamais retouché. L'algorithme d'enveloppe d'origine est préservé.

6. Matrice de Couverture des Menaces

Vecteur d'Attaque Mécanisme de Neutralisation (Arsenic V2)
DoS par épuisement RAM (m_cost falsifié) Le HeaderMAC est vérifié avant tout appel Argon2id. Un en-tête modifié est rejeté immédiatement, sans allocation mémoire. Les paramètres KDF et les IDs de cipher sont couverts par ce MAC.
Troncature ou suppression de blocs Le Merkle Root scellé dans l'enveloppe chiffrée couvre la totalité des blocs. Aucun plaintext n'est écrit tant que la racine recomputée ne correspond pas.
Permutation ou injection de blocs L'index N est injecté dans le champ AAD de chaque bloc. Un bloc valide replacé à un autre index ne passera pas la vérification AEAD.
Dégradation silencieuse du cipher ou du KDF Les IDs de cipher (octets 0x07/0x08) et les paramètres Argon2id sont couverts par le HeaderMAC. Toute modification invalide l'authentification.
Réutilisation accidentelle de nonce Les nonces de blocs sont dérivés déterministiquement depuis un File_Base_Nonce aléatoire unique par fichier — la collision est structurellement impossible. L'option AES-256-GCM-SIV apporte une résistance supplémentaire à la réutilisation de nonce.
Extraction de clés (RAM) Les structures sensibles (KEK, DEK, clés de blocs) utilisent Secret<T> avec le trait Zeroize — la mémoire est effacée à la destruction.
Corruption lors d'un rekey Un backup .bak de l'en-tête original est flushé sur disque avant toute modification. Si le processus est interrompu, le magic corrompu est détecté et l'en-tête est restauré automatiquement depuis le backup.
